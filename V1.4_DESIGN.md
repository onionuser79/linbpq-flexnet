## Item #7 + #8 — CE type-6 PATH_REQ + type-7 PATH_REP (M5.3)

### Wire format (decoded from real-peer capture 2026-05-12)

Captured on iw2ohx-gw using `xnet_agent.py --monitor-cmd monitor`
(all ports) while running a 22-iteration random `D <call>` loop.
14 CE type-6/7 frames observed, all between IW2OHX-14 (xnet) and
IR3UHU-2 (a FlexNet-capable neighbor).

Example PATH_REQ (32 bytes, xnet → IR3UHU-2, asking about IR3UGM):

```
36 21 20 20 20 20 31 49 57 32 4F 48 58 2D 31 34 20 49 52 33 55 48 55 2D 32 20 49 52 33 55 47 4D
 6  ! sp sp sp sp  1  I  W  2  O  H  X  -  1  4  _  I  R  3  U  H  U  -  2  _  I  R  3  U  G  M
```

Example PATH_REP (33 bytes, IR3UHU-2 → xnet, replying with the
next hop toward IR3UGM):

```
37 26 20 20 20 20 31 49 57 32 4F 48 58 2D 31 34 20 49 52 33 55 48 55 2D 32 20 49 5A 33 4C 53 56 4D
 7  & sp sp sp sp  1  I  W  2  O  H  X  -  1  4  _  I  R  3  U  H  U  -  2  _  I  Z  3  L  S  V  M
```

### Layout (confirmed against `flexnetd/ce_proto.c:580/634/695`)

| Offset | Length   | Field        | Notes |
|--------|----------|--------------|-------|
| 0      | 1        | TYPE         | `'6'` (0x36) for REQ, `'7'` (0x37) for REP |
| 1      | 1        | HOP_BYTE     | `CE_PATH_HOP_BYTE_BASE + hop_count` = `0x20 + n` |
| 2      | 5        | QSO field    | ASCII numeric, right-justified in 5 chars. High bit of byte 0 = `CE_PATH_TRACE_BIT` flag distinguishing TRACE-kind from ROUTE-kind |
| 7+     | variable | CALLSIGN_LIST | Space-separated callsigns. REQ: `<origin> <target>`. REP: `<origin> <hop1> <hop2> ...` |

The QSO field is the load-bearing correlation key: an originator
assigns a fresh QSO when sending a REQ; the matching REP carries
the same QSO back so the originator can match async replies to
the pending query.

The HOP_BYTE in our captures was always `!` (0x21 = hop_count 1)
on REQ and varied on REP (`&` = 0x26 → 6 hops, `$` = 0x24 → 4
hops). Likely encodes "how many hops the frame has accumulated"
— increments at each forwarder.

### What v1.4.0 implements (matches flexnetd v1.0 + adds originator)

flexnetd v1.0 only implements the **target-role** half: incoming
REQ → if target matches our call, build single-hop REP, send. No
forwarding, no origination, no path cache use of incoming REPs.
Per `flexnetd/poll_cycle.c:955-1008`, the forward-when-target-
is-not-us case is gated on M2.1/M6 (not implemented).

linbpq-flexnet v1.4.0 implements **both roles** because Marco's
intent (2026-05-12) is bidirectional: a BPQ user typing `D <call>`
on linbpq must see the path (originator role), and a remote user
on another node typing `D IW2OHX-13` must get a valid reply
(target role).

**Target role** (incoming type-6, mirroring flexnetd):

```
CE_FRAME_PATH_REQ case:
  parse origin, target, qso, kind, hop_count
  if target ≈ MYCALL (case-insensitive, strip SSID for compare):
    build type-7 with single hop = MYCALL
    flex_send_frame(LINK, FLEXNET_PID_CE, reply, len)
  else:
    log "type-6 not for us, dropped (forwarding is M6)"
```

**Originator role** (new in v1.4.0, not in flexnetd v1.0):

```
BPQ D <target> command (via FlexNet_CmdDest at :1305):
  if cached path fresh (< FLEXNET_PATH_CACHE_TTL):
    render *** target (lo-hi) T=N + *** route: ...
  else:
    render *** target (lo-hi) T=N (immediate, from dest table)
    allocate fresh QSO from g_path_qso_counter (uint16, wraps)
    record pending: (qso, target, session ref, sent_time)
    send type-6 to first active FlexNet neighbor session
    ALSO send legacy L3RTT-traceroute via flex_send_l3rtt_probe
    (coexistence: whichever reply arrives first populates the
     path cache; user re-issues D to see the path)
    tell user "*** probe sent (CE6+L3RTT). Re-issue D <call> for path."
```

**On type-7 arrival** (new in v1.4.0):

```
CE_FRAME_PATH_REP case:
  parse hop_callsigns[], qso
  look up qso in g_flex_path_pending[]
  if found:
    populate FlexNetDests[probe.dest_idx].path_hops[] from hop list
    set path_updated = now
    clear pending slot
  else:
    log "unsolicited PATH_REP qso=<n>, dropped"
```

### Wire format helpers (new functions in FlexNetCode.c)

Single-source-of-truth parser, mirroring flexnetd:

```c
/* Returns 1 if PATH_REQ, 2 if PATH_REP, -1 on malformed.
   Fills out_origin, out_target/hop_callsigns, out_qso, out_kind,
   out_hop_count.  Buffers caller-owned. */
static int flex_parse_path_frame(const unsigned char *data, int len,
                                 int  *out_kind,        /* PATH_REQ=1, PATH_REP=2 */
                                 int  *out_qso,
                                 int  *out_trace,       /* 1 if TRACE bit set */
                                 int  *out_hop_count,
                                 char *out_origin,      /* MAX_CALLSIGN bytes  */
                                 char  out_hops[][FLEXNET_MAX_CALLSIGN],
                                 int  *out_n_hops);
```

Builders:

```c
static int flex_build_path_req(unsigned char *buf, int buflen,
                               int qso, int trace,
                               const char *origin, const char *target);

static int flex_build_path_rep(unsigned char *buf, int buflen,
                               int qso, int trace,
                               const char *const *hops, int n_hops);
```

### Data structures (additions)

In `asmstrucs.h` (matching twin in `FlexNetCode.c`'s gated fallback):

```c
struct FLEXNET_PATH_PROBE {
    BOOL    active;
    int     qso;                       /* 1..65535, 0 = free */
    char    target_call[FLEXNET_MAX_CALLSIGN];
    int     target_ssid;               /* -1 = any */
    int     dest_index;                /* into FlexNetDests[] */
    int     origin_session_idx;        /* BPQ user session that issued D */
    time_t  sent_time;
    BOOL    got_reply;
};
```

Globals in `FlexNetCode.c`:

```c
struct FLEXNET_PATH_PROBE FlexNetPathProbes[FLEXNET_MAX_PATH_PROBES];
static uint16_t g_path_qso_counter = 0;   /* incremented before use */
```

`FLEXNET_MAX_PATH_PROBES = 8` (more than the 4 used for L3RTT —
expect more concurrent path probes during heavy D-query usage).

QSO allocation: increment counter, skip 0, check no in-flight
probe uses the same QSO (linear scan of 8 entries — fine).

### CE frame dispatch updates

`flex_parse_ce_frame` at `:1633` returns `CE_FRAME_DEST_BCAST=8` for
byte `'8'` (0x38). Need to add:

```c
case '6': return CE_FRAME_PATH_REQ;   /* new */
case '7': return CE_FRAME_PATH_REP;   /* new */
```

`FlexNet_ProcessCE` switch gets two new cases handling the actions
described above.

### BPQ D command output format (matches xnet)

Already implemented in `flex_show_dest_detail` (`:1265`). The
existing format is:

```
*** IR5S   (0-15) T=3
*** route: IW2OHX-13 IW2OHX-14 IR3UHU-2 IQ5KG-7 IR5S
```

which already mirrors xnet's format. No change to the output
function — only to what fills `e->path_hops[]` (now CE type-7 or
L3RTT, whichever arrives first).

### Coexistence with L3RTT-traceroute (per Marco's decision)

Both mechanisms fire on D `<call>`:

1. CE type-6 to first active FlexNet neighbor (fast path — xnet's
   native mechanism)
2. L3RTT-traceroute via `flex_send_l3rtt_probe` (fallback — works
   with older / different peers)

Both `flex_check_probe_reply` (existing, for L3RTT-traceroute) and
the new `CE_FRAME_PATH_REP` case populate the same
`FlexNetDests[dest_idx].path_hops[]` cache. The user re-issues D
and sees whichever arrived first.

If type-6 doesn't reach the right peer (e.g. neighbor doesn't
implement type-6), L3RTT-traceroute provides backstop. If
L3RTT-traceroute doesn't reach (e.g. xnet ignores it as it did
in our capture), type-6 wins. Either way the user sees a path.

### Code surface

- `FlexNetCode.c`: ~180 lines net add (parser, two builders,
  pending table, two new case branches, originator hook in
  `FlexNet_CmdDest`)
- `asmstrucs.h`: `FLEXNET_PATH_PROBE` struct + `FlexNetPathProbes[]`
  extern decl
- No `Cmd.c` changes (FlexNet_CmdDest is in FlexNetCode.c already)
- Version bump: `FLEXNET_VERSION_STR "v1.3.7"` → `"v1.4.0"`

### Out of scope for v1.4.0

- **Cross-port forwarding** when target ≠ MYCALL (M2.1/M6 in
  flexnetd's milestone notation). Drop incoming type-6 with
  log line.
- **TRACE kind** (high-bit set on QSO field byte 0). We can
  detect it but our reply is identical to ROUTE-kind.
- **Path cache resize** (item #9). Current 16-hop, 120s TTL
  per-dest stays. Resize lands in v1.4.1.
- **Background path probing** (item #10). On-demand from D
  command only. v1.4.2 problem.
- **SSID encoding in PATH_REP hop names** — our captured reply
  shows `IZ3LSVM` (7 chars) where the human-readable form is
  `IZ3LSV-14`. Likely SSID encoded as single byte (= chr(0x4D) =
  'M'). Implementation will store/echo as-received; pretty-print
  conversion is a v1.4.x polish item.

### Test plan

**Target role** (incoming type-6 we should respond to):

The verification capture for v1.3.7 showed xnet emits type-6 only
to peers it has reason to query (specifically IR3UHU-2 for
destinations beyond it). xnet never type-6'd us because nothing in
its routing table flagged us as the path-to-anywhere.

To force xnet to type-6 us, we need one of:

1. A user on a remote node (e.g. IR5S) types `D IW2OHX-13`. The
   query propagates back to xnet, which type-6s us about ourselves.
   Requires Marco's manual action from a remote console.
2. Add a dest entry in linbpq's table for a fake call (e.g.
   `IW2OHX-99`) reachable only via us, advertise to xnet,
   then have xnet's user type `D IW2OHX-99`. xnet's only-route-
   to-it is via us → type-6.

For initial verification, Marco does option 1 from a console he
has access to. We watch `/tmp/flexnet_axudp.log` for
`PATH-REQ-RX` and `PATH-REP-TX` log lines.

**Originator role** (BPQ D command on linbpq):

We telnet into linbpq's BPQ console (`iw2ohx-bpq` is Windows-only;
need to find or set up a linbpq local console). Issue `D IR5S`.
Watch debug log for `PATH-REQ-TX` event and the eventual
`PATH-REP-RX`. Re-issue `D IR5S` and confirm the path is rendered
in the BPQ session output.
