# v1.3 — Protocol correctness: design notes

Living document for the `v1.3-protocol-correctness` branch. One section per
roadmap item, filled in just before that item's implementation. Aim: catch
design mistakes before code lands, not after.

## Item #1 + #2 — L3RTT c1-c4 counters and link-down guard

### Current behavior (v1.2.0)

`FlexNetCode.c:651-666` — when a CF frame starting with `"L3RTT:"` arrives,
we either:

1. Match it against pending probes (`flex_check_probe_reply`) — used to
   compute outbound RTT; or
2. Echo the frame **byte-for-byte** back to the sender via `flex_send_frame`.

The echo path is the bug. Peers expect us to **regenerate** the L3RTT frame
with our own `c1..c4` counters so they can compute RTT from our reply. Echoing
their bytes back gives them no timing information.

### Target behavior (matches flexnetd `cf_proto.c:37-65` + `poll_cycle.c:1174-1220`)

When an incoming L3RTT probe arrives, we build a fresh reply:

```
L3RTT:<c1><c2><c3><c4> <alias> LEVEL3_V2.1 <version> $M<max_dest> $N\r
       11d  11d  11d  11d
```

Counter semantics (FlexNet protocol, 10 ms ticks):

| Field | Meaning | Our value when replying |
|-------|---------|-------------------------|
| c1    | Echo of peer's c1 (their send tick) | parsed from incoming probe |
| c2    | Echo of peer's c2 (their recv tick at last hop) | parsed from incoming probe |
| c3    | Our recv tick (when we got their probe) | local tick at frame arrival |
| c4    | Our send tick (when we emit this reply) | local tick at frame emission |

**Link-down guard (item #2):** if our local routing table has zero reachable
destinations, set `c3 = 0` and `c4 = 0`. Peer interprets this as "we're up
but our link is down" and removes routes that go through us. This is a single
conditional on a `flex_count_reachable()` helper.

### What needs to change

#### 1. Tick source — `FlexNet.h` + new helper

We need a 10 ms-resolution monotonic counter. Today the code uses `time(NULL)`
(1 s resolution) for keepalives and probe timeouts; that's too coarse for
RTT.

Add a portable helper (Linux-only target — LinBPQ runs on Debian iw2ohx-gw):

```c
/* in FlexNetCode.c (static) — returns monotonic ticks since first call,
   wraps at 2^32 ticks ≈ 497 days, matches flexnetd convention */
static uint32_t flex_get_ticks_10ms(void);
```

Internally: `clock_gettime(CLOCK_BOOTTIME, &ts)` with caching of the boot-time
epoch on first call. Falls back to `CLOCK_MONOTONIC` if BOOTTIME is unavailable
on older kernels.

#### 2. Counter parser — new function

Parse incoming `"L3RTT:%11lu%11lu%11lu%11lu ..."` into four `uint32_t`:

```c
static int flex_parse_l3rtt(const unsigned char *data, int len,
                            uint32_t *c1, uint32_t *c2,
                            uint32_t *c3, uint32_t *c4);
```

Returns 0 on success, -1 on malformed. Strict on length: the four counters
are fixed-width 11-char decimal fields per the wire format.

#### 3. Counter builder — replaces the echo path

Mirror `cf_proto.c:cf_build_l3rtt()` but adapted to LinBPQ types:

```c
static int flex_build_l3rtt(unsigned char *buf, int buflen,
                            uint32_t c1, uint32_t c2,
                            uint32_t c3, uint32_t c4,
                            const char *alias, uint32_t max_dest);
```

Format: `"L3RTT:%11lu%11lu%11lu%11lu %-6.6s LEVEL3_V2.1 %s $M%u $N\r"`.
Version string can be a `#define FLEXNET_L3_VERSION_STR "linbpq-1.3"` —
this surfaces in peer logs and helps with field debugging.

#### 4. Reachable-count helper — for link-down guard

We need the count of dest entries in our table whose RTT > 0 and which
aren't marked infinity:

```c
static int flex_count_reachable(void);
```

Iterates `FlexNetDestTable[]` (existing structure used by `dtable_merge`).
Linear scan over ≤256 entries is fine — called only on L3RTT reply path.

#### 5. CF processor rewrite — `FlexNetCode.c:651-666`

Replace the echo branch with:

```c
if (len >= 6 && memcmp(data, "L3RTT:", 6) == 0) {
    int handled = flex_check_probe_reply(data, len, LINK);
    if (handled) { /* unchanged */ }
    else {
        /* incoming probe — build a real reply, do NOT echo */
        uint32_t peer_c1, peer_c2, peer_c3, peer_c4;
        if (flex_parse_l3rtt(data, len, &peer_c1, &peer_c2,
                             &peer_c3, &peer_c4) == 0) {
            uint32_t recv_tick = flex_get_ticks_10ms();
            int reachable     = flex_count_reachable();
            uint32_t reply_c3 = (reachable > 0) ? recv_tick : 0;
            uint32_t reply_c4 = (reachable > 0) ? flex_get_ticks_10ms() : 0;
            unsigned char reply[256];
            int rlen = flex_build_l3rtt(reply, sizeof(reply),
                                        peer_c1, peer_c2,
                                        reply_c3, reply_c4,
                                        MYALIAS, FlexNetDestTableCount);
            if (rlen > 0)
                flex_send_frame(LINK, FLEXNET_PID_CF, reply, rlen);
        }
        /* else: malformed probe — silently drop */
    }
}
```

`MYALIAS` is the existing global; `FlexNetDestTableCount` is the existing
counter incremented in `dtable_merge`.

### Out of scope for this commit

- IIR filter on `our_link_time` — that's item #5; for now keep the
  hardcoded `sess->our_link_time = 2` so we land #1+#2 in isolation
- Updating outbound probes (`flex_send_l3rtt_probe`) to use 10 ms ticks
  instead of seconds — defer to #5, which is the consumer of the new
  precision

### Test plan (manual, on iw2ohx-gw)

1. Build via `sync-and-build.sh`, restart linbpq
2. From a peer (xnet IR5S, then flexnetd on the same box if available),
   trigger `n` (nodes) to fire L3RTT probes
3. Capture the wire frames with `tcpdump -i ax0 -X` — verify our reply
   has non-zero c3/c4 and well-formed counter fields
4. Disable all our routes (`r delete *`?) and re-trigger — verify our
   reply now has `c3 = 0 c4 = 0`
5. Sanity: peer's nodes table should compute a finite RTT to us
   (currently it shows `??` because no counters)

### Open questions

1. **Tick wraparound at 2^32**: 497 days. flexnetd doesn't handle wrap
   explicitly because RTT calc subtracts and ignores absolute value.
   Confirm peer's RTT calc does the same.
2. **`max_dest` value**: flexnetd passes a count it tracks itself.
   Should we pass `FlexNetDestTableCount` or the configured max?
   flexnetd source suggests the live count — go with that.
3. **Alias source**: `MYALIAS` is the BPQ alias, max 6 chars. Do we
   pad/truncate ourselves or rely on the `%-6.6s` snprintf format?
   Answer: format handles it.

## Item #1+#2 follow-up — Option B: NetRom L3 envelope on outbound replies

### Why this is needed (capture evidence, 2026-05-10)

After deploying `e60533f` (raw `CLOCK_MONOTONIC` tick source), a 55-minute
xnet_agent capture between IW2OHX-13 (linbpq) and IW2OHX-14 (xnet) produced:

| Side | Observation |
|------|-------------|
| linbpq `/tmp/flexnet_axudp.log` | 11 well-formed `L3RTT-TX` events, 5-min cadence, every reply with non-zero `our_c3`/`our_c4`, `reachable=200`, `version=linbpq-1.3` |
| xnet capture `monitor_frames` | 11 incoming L3RTT probes from IW2OHX-14 → IW2OHX-13 (1:1 with our replies) |
| xnet `L*` snapshot for `2:IW2OHX-13` | F-row rtt: `1` for 10 min, then a single transition to `2`, then **stuck at 2** for 45 min and 10 more probes |
| xnet `L*` Q-row | Outstanding probes grew `7 → 9` and never drained |

The Q-row growth and the frozen F-rtt mean xnet **receives** our replies but
does not **bind** them to its pending-probe table — so it never marks the
probe as answered and never re-samples rtt. The L3RTT payload is correct;
the **NetRom L3 envelope** xnet uses to match reply ↔ probe is missing on
our outbound.

Today the reply path emits only:
```
[PID=0xCF] [L3RTT: %11lu%11lu%11lu%11lu ... \r]
```
xnet expects:
```
[PID=0xCF] [NET/ROM L3 hdr 20 bytes, opcode=INFO] [L3RTT: ... \r]
```

### What we already have (no new infrastructure needed)

`flexnet_l3.h/c` (in this repo, used by both linbpq-flexnet and flexnetd)
exposes everything we need:

```c
int flexl3_parse(unsigned char *data, int len, struct FLEXL3_HEADER *hdr);

int flexl3_build_info(unsigned char *buf, int buflen,
    unsigned char *dest,    unsigned char *source,
    int ttl, int index, int id,
    int tx_seq, int rx_seq,
    unsigned char *payload, int payload_len);
```

The header is fixed-layout 20 bytes (`flexnet_l3.c:47-106`):

```
0..6   source callsign (AX.25 shifted)
7..13  destination callsign (AX.25 shifted)
14     TTL
15     circuit index   (IN)
16     circuit ID      (ID)
17     tx_seq          (S(n))
18     rx_seq          (R(n))
19     opcode          (low 4 bits: 1=CREQ … 5=INFO … 6=IACK)
```

L3RTT probes are carried as opcode=`FLEXL3_INFO` (5).

### Target reply path

In `FlexNetCode.c:893` (the v1.3 reply branch added for items #1+#2):

```c
/* current (v1.3.0) — bare payload */
unsigned char reply[256];
int rlen = flex_build_l3rtt(reply, sizeof(reply),
                            peer_c1, peer_c2, reply_c3, reply_c4,
                            MYALIASTEXT, FlexNetDestCount);
if (rlen > 0)
    flex_send_frame(LINK, FLEXNET_PID_CF, reply, rlen);
```

After Option B:

```c
/* 1. Try to read the incoming L3 envelope so we can mirror IN/ID. */
struct FLEXL3_HEADER in_hdr;
int have_l3 = (flexl3_parse((unsigned char *)data, len, &in_hdr) == 0
               && in_hdr.opcode == FLEXL3_INFO);

/* 2. Build the L3RTT payload exactly as today. */
unsigned char l3rtt[256];
int plen = flex_build_l3rtt(l3rtt, sizeof(l3rtt),
                            peer_c1, peer_c2, reply_c3, reply_c4,
                            MYALIASTEXT, FlexNetDestCount);
if (plen <= 0) { /* drop, log */ return; }

/* 3. Wrap in a matching L3 INFO frame if we had an envelope to mirror;
 *    otherwise fall back to the bare payload (preserves v1.2 behaviour
 *    against any peer that sends raw L3RTT). */
unsigned char frame[FLEXL3_MAX_FRAME];
int flen;
if (have_l3) {
    flen = flexl3_build_info(frame, sizeof(frame),
        in_hdr.source,                      /* dest = peer (swap)            */
        node_callsign_ax25,                 /* source = our node call        */
        FLEXL3_DEFAULT_TTL,                 /* fresh TTL, not decremented    */
        in_hdr.circuit_index,               /* IN echo — key matching field  */
        in_hdr.circuit_id,                  /* ID echo — key matching field  */
        0, 0,                               /* S/R = 0 for connectionless    */
        l3rtt, plen);
} else {
    /* legacy / unwrapped peer: keep the existing behaviour */
    memcpy(frame, l3rtt, plen);
    flen = plen;
}
if (flen > 0)
    flex_send_frame(LINK, FLEXNET_PID_CF, frame, flen);
```

### Field-by-field decisions

| L3 field | Reply value | Rationale |
|----------|-------------|-----------|
| `source` | our node call (AX.25 packed) — `node_callsign_ax25` | Standard NetRom L3: source identifies replying node |
| `dest`   | `in_hdr.source` (peer call) | Swap — we're returning to the originator |
| `ttl`    | `FLEXL3_DEFAULT_TTL` (20) | This is an originated reply, not a forwarded packet — start fresh, don't decrement the probe's TTL |
| `IN` (circuit_index) | echo `in_hdr.circuit_index` | **Critical** — xnet's pending-probe table is keyed on (peer, IN, ID); without echo, no match |
| `ID` (circuit_id) | echo `in_hdr.circuit_id` | **Critical** — same as above |
| `S(n)` | 0 | L3RTT is connectionless info, no sequencing |
| `R(n)` | 0 | same |
| `opcode` | `FLEXL3_INFO` (5) | set by `flexl3_build_info` internally |

The `IN`/`ID` echo is the load-bearing change. `source`/`dest`/`TTL`
are conventional and standard NetRom; they're correct but not what was
breaking the match.

### What needs to change in code

1. **`FlexNetCode.c`** — top of file: `#include "flexnet_l3.h"` (probably
   already pulled in for the existing connection state machine — verify).
2. **`FlexNetCode.c:893..930`** — replace the reply branch with the
   shape above. ~15 lines net.
3. **Source callsign source** — locate `node_callsign_ax25` (the AX.25-packed
   form of our node call). Candidates:
   - `MYCALL` (BPQ global — likely already AX.25-packed somewhere)
   - `LINK->LINKPORT->PORTCALL` (per-port — used at `FlexNetCode.c:1030`)
   - A dedicated FlexNet identity if one exists
   Confirm during implementation; pick the one that matches what we send
   on outbound CE traffic on the same link, so xnet's view of "who we are"
   stays consistent.
4. **No changes** to `flex_build_l3rtt`, `flex_parse_l3rtt_counters`, or
   `flex_get_ticks_10ms` — payload work is done.
5. **No changes** to `flexnet_l3.c` — we're consumers, not authors.

### Buffer sizing

- L3 header: 20 bytes
- L3RTT payload: ~85–95 bytes typical (4×11 counters + alias + version + `$M…$N\r`)
- Total: <120 bytes → comfortably within `FLEXL3_MAX_FRAME` (256)
- Existing `unsigned char reply[256]` → rename to `frame[256]` or use
  `FLEXL3_MAX_FRAME`; same allocation footprint.

### Out of scope for this commit

- Any change to `flex_send_l3rtt_probe` (we don't originate probes —
  see roadmap #5). xnet → linbpq direction only.
- Touching `flexnet_l3.c` or its connection state machine.
- Logging the wrapped frame to `/tmp/flexnet_axudp.log` differently
  (the `L3RTT-TX:` line continues to log just the counters; the wrap
  is plumbing).

### Test plan (manual, on iw2ohx-gw)

1. **Pre-deploy capture** — rerun `xnet_agent.py --duration 1800
   --snapshot-interval 60` against current build to lock the
   "Q grows / rtt stuck at 2" baseline as a control.
2. **Optional: tcpdump verification of byte layout.**

### Open questions

1. **Does xnet's pending-probe table match by `(IN, ID)` or by
   `(source_call, IN, ID)`?** If the latter, source must match
   what xnet expects — verify `node_callsign_ax25` choice in §3.
   Easy to test: if rtt still stuck after deploy, try a different
   source call source.
2. **TTL: fresh `FLEXL3_DEFAULT_TTL` or `in_hdr.ttl`?**
   Conventional NetRom replies use a fresh TTL. Picked that. If
   xnet rejects, fall back to mirror-then-decrement.
3. **Should `dest` be peer's call or the literal `"L3RTT  "` pseudo-call**
   (which is what xnet uses for probe `dest`)? Probably peer's call —
   replies are addressed, probes are quasi-broadcast — but worth
   eyeballing one of flexnetd's wire replies to confirm.

### Resolution (2026-05-10, three iterations)

Three captures on iw2ohx-gw against IW2OHX-14 (xnet) converged on the
final field choices:

**v1.3.1 — bare-payload baseline shipped with the design doc** (`dest=peer`,
`ttl=FLEXL3_DEFAULT_TTL=20`).
Result: xnet's monitor classified our replies as L3RTT for the first time
(5 captured), Q-row drained 8→5 once. But F-row rtt frozen at 1 — wrap
parsed but not bound consistently. Probe arrived with `LT 2`, our reply
went out with `LT 20`. Hypothesis: low TTL is xnet's L3RTT class marker.

**v1.3.2 — mirror everything from the probe** (`dest=in_hdr.dest` =
"L3RTT" pseudo, `ttl=in_hdr.ttl`).
Result: 35 L3RTT frames in 22 min (3.5× v1.3.1), Q-row vanished entirely.
But the capture revealed a **forwarding loop**: 11 frames showed AX.25
src=`IW2OHX-14` while their L3 src was still `IW2OHX-13` and TTL had
decremented 2→1→0. The "L3RTT" pseudo-callsign is a multicast group
in xnet's L3 forwarder; sending our reply with `dest=L3RTT` made xnet
re-broadcast it to every L3RTT subscriber, including us. Open Question
#3 answered: **never mirror `dest` to the L3RTT pseudo.**

**v1.3.3 (final, this release) — `dest=peer` + `ttl=in_hdr.ttl`.**
Combines v1.3.1's unicast `dest` (no loop) with v1.3.2's TTL mirror
(L3RTT class marker preserved). The TTL mirror is the load-bearing
discriminator we needed; the `dest` mirror was a red herring.

### Final field-by-field decisions (post-resolution)

| L3 field | Reply value | Source of truth |
|----------|-------------|-----------------|
| `source` | `MYCALL` (7 bytes AX.25-packed) | `FlexNetCode.c:119` global, also used at line 1761 for FlexNet identity |
| `dest`   | `in_hdr.source` (peer's call, unicast) | v1.3.2 loop showed L3RTT pseudo is multicast — never use it for `dest` |
| `ttl`    | `in_hdr.ttl` (echo, no decrement) | xnet's L3RTT class marker — without it, reply isn't routed to xnet's L3RTT handler |
| `IN`     | `in_hdr.circuit_index` echo | xnet's pending-probe table key |
| `ID`     | `in_hdr.circuit_id` echo | xnet's pending-probe table key |
| `S(n)`   | 0 | connectionless |
| `R(n)`   | 0 | connectionless |
| `opcode` | `FLEXL3_INFO` (5) | set by `flexl3_build_info` internally |

### F-row rtt observation across all three variants

`2:IW2OHX-13` row's F-rtt was `1` (10 ms displayed floor) in **every**
snapshot of every capture, including v1.3.3. Originally treated as a
bug indicator; now understood as the converged value for local AXUDP
between IW2OHX-13 and IW2OHX-14 on the same Raspberry Pi (sub-10 ms
actual RTT). Multiple other long-uptime neighbors in the same captures
(8:IQ2LB, 6:IW2OHX-1, 1:IW2OHX-12, 11:IR3UHU-2) also sit at `1`.
**rtt=1 is not a sign of brokenness on this link.**

The protocol-binding success criterion is **Q-row drainage**, not F-rtt
variance. v1.3.3 makes Q-row essentially absent (probes matched and
removed before each snapshot reads `L*`), which is the qualitative win
over v1.1 (Q grew 7→9) and v1.3.1 (Q churned 0/5/6/8).

## Item #5 — Link-time IIR filter

### Current behavior (v1.3.3)

`sess->our_link_time` is seeded to `2` (200 ms wire) at session start
(`FlexNetCode.c:563`) and never changes. The value is embedded as `1<n>\r`
in every outbound CE link-time frame at three TX sites:

- `:718` — LT sent in response to peer's keepalive
- `:757` — LT sent in reply to peer's LT
- `:1053` — LT sent by the proactive 21 s KA timer in `FlexNet_Tick`

`flex_get_ticks_10ms()` (added in v1.3.0) gives us a raw `CLOCK_MONOTONIC`
counter at 10 ms resolution — the precision needed to measure CE LT
round-trips on local AXUDP. Without #5 it's only used on the L3RTT reply
path. #5 wires it into the LT-TX path.

### Sample source — decision: CE link-time round-trip (Option A)

Two options were on the table:

- **A) CE LT round-trip.** Stamp tick when we TX a CE LT frame; on the
  next CE LT RX from the same peer, the delta is the link round-trip.
  Mirrors flexnetd's `g_link_stats.rtt_smoothed` machinery at
  `poll_cycle.c:864-868`. ~15 lines on top of the existing LT flow.
- **B) L3RTT timing-probe round-trip.** Originate counter-bearing
  L3RTT probes ourselves, consume peer's c4-c3 reply for 10 ms-tick
  precision samples. Requires a full originator (pending-probe table
  keyed on `(peer, IN, ID)`, scheduling) — v1.4-shaped work.

**Picked A.** Matches the stated v2.0 GA parity goal, lands in one
focused commit, and the sample cadence (~189 s with xnet's native KA;
21 s on our proactive timer when peer is silent) is fine for a 3:1 IIR.
B remains a candidate for v1.4 once an L3RTT originator exists alongside
M5 path-discovery work.

### Target behavior (matches flexnetd `poll_cycle.c:864-868`)

Standard exponential weighted moving average, weight 3:1, seeded with
the first sample:

```c
if (lt_sample_count == 0)
    lt_smoothed_10ms = sample;                                 /* seed */
else
    lt_smoothed_10ms = (lt_smoothed_10ms * 3 + sample) / 4;    /* IIR */
```

flexnetd uses the same formula (`rtt_smoothed = (rtt_smoothed * 3 +
rtt_ticks) / 4`) with a seed gate of `rtt_smoothed == 0 &&
keepalive_count <= 1`. Using an explicit `lt_sample_count` counter is
the same semantics expressed unambiguously.

### Pairing logic

Only one outstanding LT per session. The state is:

```c
uint32_t lt_tx_tick;       /* tick when we most recently sent LT      */
BOOL     lt_tx_pending;    /* TRUE between LT TX and matching LT RX   */
```

On TX: stamp tick, set pending. **Overwriting a still-pending stamp is
deliberate** — a previous LT that never got a reply is replaced by the
fresh one as the new pairing anchor. This loses one sample but avoids
spurious huge deltas after a transient drop.

On RX (CE_FRAME_LINK_TIME case): if `lt_tx_pending`, compute
`sample = (uint32_t)(now_tick - lt_tx_tick)` (wrap-safe uint32
subtraction; the 32-bit tick counter wraps every ≈497 days), fold into
IIR, clear pending. **Measure before replying** — the reply itself
re-arms the pending stamp.

### Sample filter — 60 s ceiling

Discard `sample == 0` (clock granularity edge case) and `sample > 6000`
(60 s in 10 ms ticks). 60 s comfortably exceeds the longest legitimate
exchange (xnet's 189 s KA only triggers our reply, not the round-trip;
our 21 s proactive KA elicits a reply within seconds on a healthy link)
and catches mis-pairing if a peer sends an unsolicited LT well after
our pending one expired in their view.

### Wire conversion + clamp

`our_link_time` stays in 100 ms wire units (no caller changes); internal
IIR state stays in 10 ms ticks (no precision loss across samples):

```c
uint32_t wire = (lt_smoothed_10ms + 5) / 10;   /* round to nearest */
if (wire < 1)    wire = 1;
if (wire > 4095) wire = 4095;                  /* PCFlexnet 12-bit cap */
our_link_time = (int)wire;
```

The 4095 cap comes from PCFlexnet's `link.delay` saturation behaviour
(flexnetd `poll_cycle.c:517` comment). On local AXUDP IW2OHX-13 ↔
IW2OHX-14 (sub-100 ms RTT) the wire value will sit at `1` — the same
as the hardcoded baseline; #5's win is that it now *tracks* the link
instead of asserting a guess.

### Session-state additions

```c
struct FLEXNET_SESSION {
    /* … existing fields … */
    uint32_t lt_smoothed_10ms;   /* IIR state in 10 ms ticks            */
    uint32_t lt_tx_tick;         /* tick of most recent LT TX           */
    uint32_t lt_sample_count;    /* samples folded in (0 = uninit/seed) */
    BOOL     lt_tx_pending;      /* TRUE between LT TX and matching RX  */
};
```

Zero-initialised in `FlexNet_InitSession` along with the existing fields
(the `memset(sess, 0, sizeof(*sess))` at line 559 already covers this).

### Code changes

1. **TX helper** — collapse the three `flex_build_link_time` /
   `flex_send_frame` pairs at `:718,757,1053` into one helper:

   ```c
   static int flex_send_link_time(LINKTABLE *LINK,
                                  struct FLEXNET_SESSION *sess);
   ```

   Returns the byte count sent or -1. Stamps `lt_tx_tick` and sets
   `lt_tx_pending` after the send.

2. **RX sample fold** — at the top of the `CE_FRAME_LINK_TIME` case
   (before the existing `peer_link_time = atol(tbuf)` parse), call:

   ```c
   static void flex_link_time_sample(struct FLEXNET_SESSION *sess);
   ```

   No-op if `!lt_tx_pending`. Folds sample, updates `our_link_time`,

## Item #6 — `dtable_merge` RTT=0 skip

### Current behavior (v1.3.4)

`flex_dtable_merge` at `FlexNetCode.c:1726` treats every incoming COMPACT
entry uniformly: if a matching destination exists, update it when
`incoming->rtt < existing->rtt` OR `existing->port == port`. New
destinations are always inserted regardless of RTT value.

The bug bites in two places after every session init:

- xnet advertises its FlexNet dtable in **two rounds**. Round 1 carries
  measured RTTs (e.g. 242, 208). Round 2 (~20 s later) carries
  **RTT=0 for every record** — a refresh/keepalive marker for
  destinations xnet already knows.
- xnet also re-advertises back to us the routes it learned **via us**,
  also with RTT=0.

In linbpq-flexnet, RTT=0 is strictly less than any real RTT, so it
always trips the "improved" branch and overwrites the real RTT. After
~20 s the user-visible D-table is full of `RTT=0` rows.

flexnetd hit the same bug and fixed it in v0.7.5
(`dtable.c:50-75`): treat `incoming->rtt == 0 && !is_infinity` as a
protocol refresh marker, not a measurement, and skip the merge
entirely — preserve existing data if any, don't insert a new row.

### Target behavior (matches flexnetd `dtable.c:50-75`)

At the top of `flex_dtable_merge`:

```c
if (incoming->rtt == 0 && !incoming->is_infinity)
{
    /* Refresh marker, not a measurement. */
    int idx = flex_find_dest(incoming->callsign,
                             incoming->ssid_lo, incoming->ssid_hi);
    if (idx >= 0)
        FlexNetDests[idx].last_updated = time(NULL);   /* touch */
    /* else: nothing to do — the destination will re-advertise with a
       real RTT later. */
    return 0;
}
```

`!is_infinity` is the load-bearing guard. Withdrawn-route broadcasts
arrive with `rtt = FLEXNET_RTT_INFINITY (60000)` AND `is_infinity = 1`
(see the existing `flex_parse_compact_records` at `:1696`); the RTT=0
check must not swallow those. With the `!is_infinity` clause the
guard fires only on the genuine `RTT=0, is_infinity=0` refresh case.

`flex_find_dest` doesn't yet exist as a named helper — currently the
linear scan is inlined in `flex_dtable_merge`. Two options:

1. Extract `static int flex_find_dest(call, lo, hi)` and call it from
   both the new skip path and the existing merge path. ~10 lines net
   refactor, slight overall simplification.
2. Inline a second linear scan in the skip path. Faster to write,
   adds ~6 lines of duplication. **Picked option 1** — the helper is
   already used implicitly in three other places (line 631, 1343, 1428
   patterns) that would also benefit from a shared lookup.

### Caller-side cosmetic update (`FlexNetCode.c:817`)

The COMPACT-batch caller logs each entry's tag based on `rc`:

```c
const char * tag = (entries[i].rtt >= FLEXNET_RTT_INFINITY)
                   ? "withdrawn" : (rc == 1 ? "new" : "updated");
```

With #6, an `rc == 0, rtt == 0, !is_infinity` entry would fall through
to `"updated"` in this expression — a misleading FLEXNET_DEBUG log
line. Add an explicit `"skip (refresh)"` branch and a `skip_cnt`
counter in the batch summary. Pure debug-side change, no protocol
effect.

### Return code semantics

Today linbpq-flexnet's `flex_dtable_merge` returns:
- 1 = new
- 2 = updated (or no-op same-rtt match — existing minor wart, not
  fixing in this commit)
- 0 = table full

flexnetd returns 0 for both skipped and no-change. Keeping linbpq's
0 = skipped/table-full conflation is fine for now — the COMPACT-batch
caller only branches on `rc == 1` and `rc == 2` for counters. A
dedicated rc value for "skipped (refresh)" would be cleaner but
unnecessary for the protocol fix.

## Item #3 — Keepalive interval 21s → 180s

### Current behavior (v1.3.5)

`FlexNet_Timer` at `FlexNetCode.c:1062` fires the proactive keepalive
when `now - sess->last_keepalive >= 21`. `last_keepalive` is updated
whenever we send OR receive a KA (the `CE_FRAME_KEEPALIVE` case at
`:731` and the timer itself at `:1078`).

Against xnet's native 189 s KA cadence, this means: per 189 s window,
1 xnet KA arrives → we echo (reactive). Then our 21 s timer fires
~8 times before xnet's next KA → 8 proactive KAs. Plus 1 reactive
echo on the next xnet KA → ~10 KAs per 189 s window.

`flex_build_keepalive` produces a 241-byte payload (CE marker + 240
padding spaces). At 10 KAs per 189 s, that's roughly **15 kB/min of
KA traffic per session**, multiplied by the route advertisements
that piggyback after the keepalive. On a 1200 baud or 9600 baud
AX.25 link this is significant overhead, on AXUDP it's wasted UDP
volume and CPU.

### Target behavior

Match the FlexNet PROTOCOL_SPEC cadence: send proactive keepalives
at **180 s** intervals, not 21 s. One-line constant change at
`:1062`.

Coexistence with xnet's 189 s native cadence:

- t=0 — xnet KA arrives, we echo, `last_keepalive=0`.
- t=180 — proactive timer fires (now-0 >= 180), we send KA + LT,
  `last_keepalive=180`.
- t=189 — xnet's next KA arrives, we echo, `last_keepalive=189`.
- Next proactive: t = 189 + 180 = 369. xnet's next KA at t=378.
- Per 189 s window: 1 xnet KA + 1 our proactive + 1 reactive echo
  = 3 KAs. ~70% reduction from today's 10.

The proactive still fires once per xnet cycle (the 9 s gap between
180 and 189). With #4 (300 s adaptive threshold) the proactive
becomes truly silent on an active xnet link — but that's #4's job.
For #3 alone, the win is "8 redundant KAs per cycle eliminated."

### Code changes

Single literal change at `:1061-1062`:

```c
/* Send keepalive every 180 seconds (PROTOCOL_SPEC). On an active
   xnet link (189 s native cadence) this fires once per xnet cycle
   in the 9 s gap before xnet's next KA arrives — #4 will replace
   this with an adaptive 300 s threshold so the proactive stays
   silent on a busy link. */
if (now - sess->last_keepalive >= 180)
```

No struct changes, no other call sites affected, no asmstrucs.h
changes.

### v1.3.4 IIR wire-advertisement regression — fix folded into v1.3.6

**Observed during v1.3.6 mid-capture monitoring** (2026-05-11): xnet's
`L*` Q/T column for `2:IW2OHX-13` shows T=65 (stuck), while every
healthy peer sits at T=1–3:

```

9:DK0WUE      F  2
11:IR3UHU-2   F  1
13:IR1UAW-10  F  3
2:IW2OHX-13   F  65   ← us — outlier
```

xnet's T column is its smoothed view of **the peer's advertised CE
link-time value** (`"1<n>\r"` payload). v1.3.5 verification showed
linbpq-flexnet advertising values in the 100–999 range (frames at
`len=23`, i.e. 5-byte payloads with 3-digit `<n>`). Other peers
advertise 1–3, so they display T=1–3.

**Root cause: v1.3.4's #5 IIR sample source.** We sample
`now_tick - lt_tx_tick` — the CE LT round-trip — and feed it into
the 3:1 IIR. That round-trip is dominated by the peer's reply
scheduling, not network latency. We then advertise the inflated
smoothed value back to the peer, which displays it.

**flexnetd already worked around this**, per `poll_cycle.c:526-531`:
> *Link-time value carries our smoothed RTT. We hard-code 2 (= 200 ms
> wire) because our actual smoothed measurement is not yet
> implemented; this matches linbpq-flexnet's value and is accepted
> by both (X)Net and PCFlexnet.*

flexnetd computes smoothed RTT internally for stats but advertises
hardcoded 2 on the wire. v1.3.4 advertised the IIR output, which
was the regression.

**Fix in v1.3.6:** add `#define FLEXNET_WIRE_LT 2` and pass that
constant to `flex_build_link_time` from `flex_send_link_time`. The
IIR machinery in `flex_link_time_sample` still runs and keeps
`sess->our_link_time` up-to-date — but only for internal logging
and future use, not for the wire.

This is one-line in `flex_send_link_time`, semantically a partial
revert of #5's wire-advertisement claim but a full preservation of
the IIR sampling and smoothing infrastructure. The IIR's value is
no longer "what we tell xnet" — it's "what we know about ourselves."

#5's design doc's pass-criterion 1 ("non-constant wire LT values")
is **invalidated** by this addendum — the wire value is now back to
the constant 2. The IIR continues to verify correctly via the
debug log's `lt_sample` entries on tty2 (FLEXNET_DEBUG build).

### Correction: v1.3.5's "Q-row caveat closed" claim was wrong

The v1.3.5 commit docs claimed that v1.3.5's RTT=0 skip fix
indirectly closed the "Q-row caveat" we noticed in v1.3.4 — by

**That attribution was incorrect.** The Q-state row in xnet's

not the FlexNet link entry. xnet maintains both side-by-side:

v1.3.6 again "appears" Q.

The actual FlexNet metric (the F-row's T value) is what we should
have been tracking all along; the wire-LT fix in v1.3.6 is the
first commit that addresses it directly, and the result (T=3) is
the right answer.

"Q-row caveat closed" is **invalidated**. The Q-row was never a
v1.3.x FlexNet regression. Memory entries on v1.3.5 should be
read with this correction in mind.

## Item #4 — Proactive KA threshold 180 s → 300 s

### Current behavior (v1.3.6)

`FlexNet_Timer` fires the proactive KA when `now -
sess->last_keepalive >= 180`. `last_keepalive` is updated on every
KA, reactive or proactive (lines 731 and 1082).

Against xnet's 189 s native KA cadence the math suggests one
proactive KA fires per cycle in the 9 s gap between 180 s and
xnet's next KA at 189 s — and indeed that was the intent in #3.

**Empirical observation from v1.3.6 verification capture**, however:
the steady-state KA sequence (16:08:44, 16:11:53, 16:15:02,
16:18:11, 16:21:20) spaced at exact 189 s — **xnet's cadence,
not 180 s**. No proactive KAs visible in the trace. Reasonable
explanation: `FlexNet_Timer` is called by LinBPQ's main loop at
some coarser cadence than 1 Hz, so the 180→189 s window often
gets missed — by the time Timer runs, xnet's KA at 189 s has
already updated `last_keepalive` and the `>= 180` predicate is
no longer true.

So in *practice* the 180 s threshold already behaves like the
adaptive flexnetd target. #4 codifies the intent and removes the
9 s window of fragility: with `>= 300` the proactive timer can
**never** fire while xnet's KAs continue at 189 s cadence, because
each xnet KA resets `last_keepalive` and the 300 s threshold is
strictly larger than the 189 s interval between resets.

### Target behavior (matches flexnetd `poll_cycle.c:501`)

Single constant change at `:1066`: `>= 180` → `>= 300`.

The proactive becomes a **degraded-link backup**: it fires only
when xnet has been silent for 300 s (= ~5 min, well past xnet's
189 s natural rhythm). On a healthy link, zero proactive KAs.

This matches flexnetd's v0.7.9 reasoning exactly
(`poll_cycle.c:472-501`):
> *300 s is longer than xnet's 189 s native cadence, so xnet's
> KAs always reset `last_keepalive_tx` first and the proactive
> never fires on an active xnet link.*

flexnetd's secondary concern — that frequent proactive KAs
dominate xnet's smoothed-RTT IIR — is already addressed in
linbpq-flexnet v1.3.6 by the wire-LT decoupling (our advertised
value is hardcoded 2 regardless of how often we send LT). But
the "don't transmit unnecessarily" principle still applies, and
matching flexnetd makes the two implementations easier to reason
about side-by-side.

### Code changes

One literal at `:1066`. Comment block at `:1061-1065` updated to
reflect the new threshold rationale.

No struct changes, no other call sites affected, no `asmstrucs.h`
changes.

### Side effect on item #5 (link-time IIR)

Same as #3: `flex_send_link_time` is called inside the proactive
timer at `:1080`, so on a healthy xnet link the IIR continues to
fold one sample per ~189 s from the reactive echo path
(`CE_FRAME_LINK_TIME` case at `:737`). No regression.

### Out of scope for this commit

- **True adaptivity** (e.g. dynamically lengthening the threshold
  based on observed peer cadence). 300 s as a static constant is
  enough — xnet's cadence is well-known and stable.
- **Per-peer thresholds.** flexnetd has a `lt_reply_interval`
  config knob for PCFlexnet (320 s). linbpq-flexnet today talks
  only to xnet on the FlexNet-enabled port, so a single constant
  works. Add per-port config if/when a PCFlexnet peer is wired up.
